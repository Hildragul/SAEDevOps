Room : Putting It All Together

Contexte général de la room

Cette room sert de synthèse des concepts abordés dans les précédentes rooms : DNS, HTTP, URLs, requêtes/réponses, méthodes, cookies, et status codes.

Elle explique comment tout s’enchaîne lorsqu’on accède à un site web depuis un navigateur, et introduit aussi quelques composants supplémentaires utilisés pour optimiser le fonctionnement d’Internet.

Task 1 – Putting It All Together

Résumé de la tâche :

Quand on demande une page web, plusieurs étapes se déroulent en coulisses :

  1. Résolution DNS : pour convertir le nom de domaine en adresse IP.
  2. Requête HTTP : envoyée au serveur via un protocole structuré.
  3. Réponse du serveur : envoie les ressources (HTML, CSS, JS, images…).
  4. Affichage : le navigateur assemble les ressources pour afficher le site.

D'autres composants (comme les CDN, proxys, caches, etc.) seront vus plus loin pour montrer comment Internet devient plus rapide, plus sécurisé et plus fiable.

Task 2 – Other Components

Résumé de la tâche :

On explore ici des composants supplémentaires essentiels pour améliorer les performances, la fiabilité et la sécurité des sites web :

  • Load Balancer : Répartit les requêtes entre plusieurs serveurs pour gérer un grand trafic ou assurer une haute disponibilité. Il utilise des algorithmes (comme round-robin ou weighted) et effectue des health checks pour détecter les serveurs défaillants.
  • CDN (Content Delivery Network) : Réseau mondial de serveurs qui hébergent des fichiers statiques (images, vidéos, CSS, JS) pour les livrer plus rapidement depuis un serveur proche du client.
  • Databases : Permettent de stocker et récupérer des données côté serveur. Exemples : MySQL, MongoDB, PostgreSQL...
  • WAF (Web Application Firewall) : Analyse les requêtes HTTP pour bloquer les attaques (XSS, DDoS, bots…). Utilise des règles, du filtrage, et du rate limiting pour se défendre.

Questions et réponses :

  • Q1 : What can be used to host static files and speed up a clients visit to a website?
    R : CDN
  • Q2 : What does a load balancer perform to make sure a host is still alive?
    R : Health Check
  • Q3 : What can be used to help against the hacking of a website?
    R : WAF

Task 3 – How Web Servers Work

Résumé de la tâche :

Cette tâche explique le fonctionnement des serveurs web, leur rôle, et la distinction entre contenu statique et dynamique.

  • Un serveur web (comme Apache, Nginx, IIS, NodeJS) écoute les connexions et sert des fichiers via le protocole HTTP.
  • Les fichiers sont généralement servis depuis un dossier racine (/var/www/html sous Linux, C:\inetpub\wwwroot sous Windows).
  • Grâce aux virtual hosts, un seul serveur peut héberger plusieurs sites différents (domaines multiples).

Contenu statique vs dynamique :

  • Statique : ne change pas (images, CSS, JS, HTML fixe).
  • Dynamique : généré à la volée selon les données, les requêtes ou les utilisateurs (ex : résultats de recherche, contenu personnalisé).

Questions et réponses :

  • Q1 : What does web server software use to host multiple sites?
    R : Virtual Hosts
  • Q2 : What is the name for the type of content that can change?
    R : Dynamic
  • Q3 : Does the client see the backend code? Yay/Nay
    R : Nay

Ce que j’ai compris

Cette room m’a permis de relier tous les concepts appris précédemment sur le fonctionnement du web. Quand on accède à un site, il ne s’agit pas juste d’un clic : plein de choses se passent en arrière-plan, et chaque étape est essentielle pour que tout fonctionne correctement.

J’ai compris que tout commence par la résolution DNS, qui permet de traduire un nom de domaine en adresse IP. Ensuite, une requête HTTP est envoyée à un serveur web, qui peut être protégé par un WAF, réparti par un load balancer, ou optimisé par un CDN.

Le serveur web (comme Apache ou Nginx) va alors gérer la demande. Il peut répondre avec :

  • des fichiers statiques (images, HTML fixes...),
  • ou du contenu dynamique généré à la volée via un langage backend (comme PHP ou NodeJS), souvent en lien avec une base de données.

L’ensemble est orchestré de façon transparente pour l’utilisateur, qui ne voit que le résultat final dans son navigateur.

Enfin, j’ai mieux compris l’importance de la structure logique des requêtes, des headers, des méthodes HTTP, des status codes, et le rôle des cookies dans les sessions utilisateurs.

En résumé : cette room est une vue d’ensemble très utile pour comprendre le web dans sa globalité, avant d’aller plus loin dans l’analyse ou la sécurité des applications web.
